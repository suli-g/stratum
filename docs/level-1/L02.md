## Preamble

In Lesson 1, you saw the difference between a `Task` class that did everything and a designed system where each component had one responsibility. You understood intuitively that the first was bad. In this lesson, we give that intuition a precise vocabulary.

How do you *measure* whether a design is good? How do you explain to a teammate why splitting a class matters beyond "it feels right"? The answer is three metrics that apply to any codebase: **separation of concerns** (what responsibilities belong together), **coupling** (how much components depend on each other), and **cohesion** (how related the elements inside a component are).

By the end of this lesson, you will be able to look at a component diagram and explain why one architecture will age gracefully while another will become brittle. You will also meet three design patterns by name — Factory, Repository, and Observer — that you will encounter repeatedly throughout this course.

## §1

### Separation of Concerns

In Lesson 1, we introduced **separation of concerns** as a principle — the idea that each component should have one clearly defined responsibility, with minimal overlap into the responsibilities of others.[^1] Here we make it concrete and testable.

A violation occurs when a single component handles multiple *types* of work that could change independently. The god class in L01 §4.1 violated it: `Task` handled data storage, web display, notifications, and user assignment all at once.

```{mermaid}
classDiagram
    class Task {
        +String title
        +String description
        +String status
        +saveToDatabase()
        +formatForWebDisplay()
        +sendOverdueNotification()
        +assignToUser()
    }
```

If the notification medium changes (email → SMS), `Task` must be modified. If the display format changes (web → mobile), `Task` must be modified. If the database schema changes, `Task` must be modified. Each independent change opens the entire class to accidental breakage.

A separated design insulates those concerns from each other:

```{mermaid}
classDiagram
    class Task {
        +String title
        +String description
        +String status
    }
    class TaskRepository {
        +save(Task)
        +findById(id)
    }
    class TaskFormatter {
        +formatForWebDisplay(Task)
    }
    class NotificationService {
        +sendOverdueNotification(Task)
    }

    TaskRepository --> Task : stores
    TaskFormatter --> Task : formats
    NotificationService --> Task : reads
```

`Task` is now pure data. If the database changes, only `TaskRepository` needs to change — the formatter and notifier are completely insulated from that decision.

:::{important}
**The Single Responsibility Test.** Ask: "What change would force me to modify this component?" If you can name more than one *unrelated* reason — database changes, UI changes, notification changes — the component is carrying more than one concern.
:::

### Formative Check-in

:::{important}
Before reading on: In the separated design above, if the team decides to switch from sending overdue email notifications to sending SMS, which component changes? Why are `Task`, `TaskRepository`, and `TaskFormatter` unaffected?
:::

## §2

### Coupling

**Coupling** is the degree to which one component depends on another.[^2] Some coupling is unavoidable — components in a system must communicate. What varies is its *tightness*.

**Tight coupling** exists when a component reaches directly into the concrete implementation of another: it knows the other's field names, internal APIs, and data formats. A change to the depended-on component propagates immediately to everything that depends on it.

**Loose coupling** exists when components interact only through a well-defined, stable interface. One component does not know or care *how* the other does its work — only what it provides.

```{mermaid}
classDiagram
    class Task_Tight {
        +saveToDatabase()
    }
    class MySQLDatabase {
        +connect()
        +execute(query)
    }
    Task_Tight --> MySQLDatabase : direct dependency
```

Here, `Task` reaches directly into `MySQLDatabase`. Switching to PostgreSQL requires modifying `Task`.

```{mermaid}
classDiagram
    class Task_Loose {
        +save()
    }
    class TaskRepository {
        +save(Task)
        +findById(id)
    }
    class MySQLDatabase {
        +connect()
        +execute(query)
    }
    class PostgreSQLDatabase {
        +connect()
        +execute(query)
    }

    Task_Loose --> TaskRepository : uses interface
    TaskRepository --> MySQLDatabase : implements with
    TaskRepository ..> PostgreSQLDatabase : can swap to
```

Now `Task` depends on `TaskRepository`. The repository can swap its underlying database without `Task` knowing or caring.

:::{warning}
**Coupling disguised as convenience.** Accessing a deeply nested attribute in one step (`order.task.project.owner.email`) feels like a shortcut. It is tight coupling: your component now knows the internal structure of three other components. Any restructuring breaks your code silently.
:::

| Coupling level | Description | Change impact |
|---|---|---|
| **Tight** | Component A knows the concrete details of Component B | Changing B's internals requires changing A |
| **Loose** | Component A depends on a stable interface of Component B | Changing B's implementation does not affect A |
| **None** | Components are completely independent | No impact (rare in real systems) |

## §3

### Cohesion

**Cohesion** is the degree to which the elements within a single component belong together.[^2] High cohesion means all methods and data in a component serve one unified purpose. Low cohesion means the component has accumulated responsibilities that have little to do with each other — a god class by another name.

```{mermaid}
classDiagram
    class TaskRepository {
        +save(Task)
        +findById(id)
        +delete(id)
        +update(Task)
    }
```

`TaskRepository` has high cohesion: every method is a persistence operation on `Task` objects. A developer reading this component immediately understands its purpose and knows exactly what it will and will not do.

```{mermaid}
classDiagram
    class TaskUtils {
        +saveToDatabase(Task)
        +formatForWebDisplay(Task)
        +sendEmail(to, subject, body)
        +calculateTax(amount)
        +generatePDF(data)
    }
```

`TaskUtils` handles database operations, UI formatting, email, tax calculation, and PDF generation. These concerns are entirely unrelated. This is low cohesion.

:::{tip}
**The Cohesion Test.** Read the component name and ask: "This component handles X." If X must be a list of unrelated things — "database and email and PDFs and taxes" — cohesion is low.
:::

### Coupling and Cohesion Together

These two metrics measure opposite things but serve the same goal. Coupling is about the space *between* components; cohesion is about the space *inside* a component.

| Concept | Asks about | Goal |
|---|---|---|
| Coupling | How much does Component A depend on Component B? | Minimise |
| Cohesion | How well do the elements inside Component A belong together? | Maximise |

Their combinations produce four architectural outcomes:

| | **High cohesion** | **Low cohesion** |
|---|---|---|
| **Loose coupling** | ✅ Easy to change, easy to understand | ⚠️ Easy to change, hard to understand |
| **Tight coupling** | ⚠️ Hard to change, easy to understand | ❌ Hard to change, hard to understand |

The goal is the top-left cell: **loose coupling + high cohesion**. The bottom-right cell — the `TaskUtils` god class, tightly coupled to everything it reaches into — is the disaster scenario: every change is risky and no component is comprehensible in isolation.

:::{note}
Low cohesion and tight coupling tend to appear together. A component that does too many things inevitably reaches into the internals of many others to accomplish them. Fixing cohesion problems often resolves coupling problems at the same time.
:::

## §4

### Design Patterns: Names for Common Solutions

When you apply separation of concerns, high cohesion, and loose coupling consistently, certain structural solutions appear repeatedly. These solutions have names — **design patterns** — documented systematically in software engineering literature.[^3]

Three patterns emerge naturally from the Task Management System design. You do not need to implement them now; you need to recognise them when they appear in code.

:::{note}
**Gang of Four.** These patterns are documented in *Design Patterns: Elements of Reusable Object-Oriented Software* (Gamma, Helm, Johnson, Vlissides — 1994), commonly called the "Gang of Four" book. The names used throughout this course come from this source.
:::

#### §4.1 The Repository Pattern

The Repository pattern abstracts all data access behind a consistent interface. Instead of components querying the database directly, all storage operations flow through a repository that presents a clean API to the rest of the system.

```{mermaid}
classDiagram
    class Task {
        +String title
        +String status
    }
    class TaskRepository {
        +save(Task)
        +findById(id)
        +findAll()
        +delete(id)
    }
    class TaskService {
        +createTask(title)
        +completeTask(id)
    }

    TaskService --> TaskRepository : uses
    TaskRepository --> Task : manages
```

`TaskService` never knows whether tasks are stored in PostgreSQL, a file, or memory. The repository interface stays stable regardless of what is behind it.

#### §4.2 The Factory Pattern

The Factory pattern delegates object construction to a dedicated component. Instead of building a `Task` object inline wherever it is needed, callers ask a factory to produce it — and the factory handles default values, validation, and initialisation rules.

```{mermaid}
classDiagram
    class Task {
        +String title
        +String status
        +DateTime createdAt
        +User assignee
    }
    class TaskFactory {
        +createTask(title, assignee) Task
        +createRecurringTask(title, interval) Task
    }

    TaskFactory --> Task : creates
```

If task creation rules change — a new required field, a new validation constraint — only `TaskFactory` needs modification. All callers are insulated.

#### §4.3 The Observer Pattern

The Observer pattern defines a one-to-many dependency where a publisher notifies a set of subscribers when an event occurs, without knowing anything about them specifically.

```{mermaid}
classDiagram
    class TaskEventPublisher {
        +subscribe(event, handler)
        +publish(event, Task)
    }
    class NotificationService {
        +onTaskOverdue(Task)
    }
    class AuditLogger {
        +onTaskOverdue(Task)
    }
    class TaskMetrics {
        +onTaskCompleted(Task)
    }

    TaskEventPublisher --> NotificationService : notifies
    TaskEventPublisher --> AuditLogger : notifies
    TaskEventPublisher --> TaskMetrics : notifies
```

When `TaskEventPublisher` publishes a `task.overdue` event, every subscribed observer handles it independently. Adding a new observer (`SlackNotifier`, `BillingService`) requires no changes to the publisher or to any existing observer.

## Summary

| Concept | What you learned | Why it matters |
|---|---|---|
| Separation of concerns | Each component should have one reason to change | Changes to one concern do not break others |
| Coupling | The degree of dependency between components | Tight coupling makes every change risky |
| Cohesion | The degree to which a component's internals belong together | Low cohesion makes components hard to understand and maintain |
| Repository pattern | Abstracts data access behind a stable interface | The rest of the system is insulated from storage decisions |
| Factory pattern | Delegates object construction to a dedicated component | Callers are decoupled from construction rules |
| Observer pattern | A publisher notifies subscribers without knowing who they are | New consumers can be added without touching the publisher |

## Key Terms

- **separation of concerns**: The principle that each component should have one clearly defined responsibility, with minimal overlap into the responsibilities of others
- **coupling**: The degree to which one component depends on another
- **loose coupling**: A design where components depend on stable interfaces rather than concrete implementations — changes to one component's internals do not propagate to its dependents
- **cohesion**: The degree to which the elements within a single component belong together
- **high cohesion**: A design where all elements in a component serve one unified purpose
- **module boundary**: The defined interface between two components — the only surface through which they should communicate
- **Repository pattern**: A structural pattern that abstracts all data access behind a consistent interface, insulating the rest of the system from storage decisions
- **Factory pattern**: A creational pattern where a dedicated component is responsible for building objects, decoupling callers from construction rules
- **Observer pattern**: A behavioural pattern where a publisher notifies subscribers of events without knowing who those subscribers are
