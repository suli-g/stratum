## §1

### Separation of Concerns

The principle of **separation of concerns** states that a program should be divided into distinct sections where each section addresses a separate concern.[^1] Every component should have one reason to change. When a component mixes concerns, it has multiple reasons to change — and each change risks breaking the others.

:::{tip}
**The Single Responsibility Test**
Try to describe what a component does in a single sentence. If you find yourself using "and" or "also," the component likely has multiple concerns and should be split.
:::

In Lesson 1 we saw the `Task` god class violate this principle — it handled data storage, web formatting, user assignment, and notifications all at once. The designed version corrected this by giving each responsibility its own component:

```{mermaid}
classDiagram
    class Task {
        +String title
        +String description
        +String status
    }
    class TaskRepository {
        +saveToDatabase(Task)
    }
    class TaskFormatter {
        +formatForWebDisplay(Task)
    }
    class NotificationService {
        +sendOverdueNotification(Task)
    }

    TaskRepository --> Task : stores
    TaskFormatter --> Task : formats
    NotificationService --> Task : reads
```

Each component now has exactly one reason to change. If the notification medium changes (email → SMS), only `NotificationService` changes. If the database changes, only `TaskRepository` changes.

## §2

### Coupling: Measuring Interdependence

**Coupling** refers to the degree of direct knowledge that one component has of another.[^2] When components are **tightly coupled**, changes to one often require changes to others. When they are **loosely coupled**, components can be modified independently.

:::{warning}
**Coupling Disguised as Convenience**
Tight coupling often appears as a shortcut. Directly instantiating a dependency inside a component feels faster — but it binds the two components together permanently, making testing and future changes expensive.
:::

Compare two designs for `TaskRepository`. In the first, the repository creates its own database connection:

```{mermaid}
classDiagram
    class TaskRepository {
        -connection : MySQLConnection
        +TaskRepository()
        +save(Task)
    }
    class MySQLConnection {
        +connect(host, user, password)
    }
    TaskRepository --> MySQLConnection : creates internally
```

In the second, the connection is *passed in* rather than created:

```{mermaid}
classDiagram
    class TaskRepository {
        -connection : DatabaseConnection
        +TaskRepository(connection)
        +save(Task)
    }
    class DatabaseConnection {
        <<interface>>
        +cursor()
    }
    TaskRepository --> DatabaseConnection : depends on abstraction
```

In the loosely coupled design, `TaskRepository` does not care whether the connection is MySQL, PostgreSQL, or a test fake. It depends on an abstraction, not a concrete implementation. This is called **dependency injection** — the dependency is provided from outside rather than created inside.

## §3

### Cohesion: Measuring Focus

**Cohesion** measures how closely related the responsibilities within a single component are.[^3] **High cohesion** means a component does one thing well. **Low cohesion** means a component tries to do many unrelated things.

:::{note}
High cohesion and low coupling tend to go together. A component that does one thing well (high cohesion) typically needs fewer dependencies on other components (low coupling).
:::

Consider a catch-all utility component in a task management system:

```{mermaid}
classDiagram
    class TaskUtilities {
        +calculateOverdueFee(Task)
        +formatForWebDisplay(Task)
        +sendNotification(Task)
        +validateTaskTitle(Task)
    }
```

This component has **low cohesion** — it mixes fee business logic, presentation logic, external communication, and data validation. No single sentence can describe what it does without using "and."

The high-cohesion alternative splits each responsibility into its own component:

```{mermaid}
classDiagram
    class FeeCalculator {
        +calculateOverdueFee(Task)
    }
    class TaskFormatter {
        +formatForWebDisplay(Task)
    }
    class NotificationService {
        +sendNotification(Task)
    }
    class TaskValidator {
        +validateTaskTitle(Task)
    }
```

Each component now has a single, clearly-stated purpose. Each is independently testable and reusable.

## §4

### Design Patterns: Named Solutions

When separation of concerns, low coupling, and high cohesion are applied consistently, certain structures emerge again and again across codebases. These recurring structures have been catalogued and named — they are called **design patterns**.[^4]

You do not need to implement these patterns yet. The goal here is recognition: when you see a pattern in code later in this course, you will know its name and what problem it is solving.

#### The Repository Pattern

Separates data persistence from business logic. The rest of the application talks to a repository, not directly to a database.

```{mermaid}
classDiagram
    class TaskRepository {
        +save(Task)
        +findById(id)
        +findAllOverdue()
    }
    class Task {
        +String title
        +String status
    }
    TaskRepository --> Task : manages
```

#### The Factory Pattern

Separates object creation logic from the rest of the application. When constructing an object becomes complex, a factory component owns that complexity.

```{mermaid}
classDiagram
    class TaskFactory {
        +createUrgentTask(title) Task
        +createScheduledTask(title, dueDate) Task
    }
    class Task {
        +String title
        +String priority
        +String status
    }
    TaskFactory --> Task : creates
```

#### The Observer Pattern

Separates event producers from event consumers. A component announces that something happened; other components decide independently what to do about it.

```{mermaid}
classDiagram
    class TaskEventPublisher {
        +onTaskOverdue(Task)
    }
    class NotificationService {
        +handleOverdueEvent(Task)
    }
    class AuditLogger {
        +handleOverdueEvent(Task)
    }
    TaskEventPublisher --> NotificationService : notifies
    TaskEventPublisher --> AuditLogger : notifies
```

## Summary

| Concept | What you learned | Why it matters |
|---|---|---|
| Separation of concerns | Each component should have one clearly defined responsibility | Prevents components from becoming brittle and hard to change |
| Coupling | The degree to which one component depends directly on another | Loosely coupled components can be changed and tested independently |
| Cohesion | How focused and related the responsibilities within a component are | Highly cohesive components are easier to understand, test, and reuse |
| Design patterns | Named, recurring solutions to common structural problems | Allow you to recognise and communicate architectural intent quickly |

## Key Terms

- **separation of concerns**: The principle that each component should have one clearly defined responsibility, with minimal overlap into the responsibilities of others
- **coupling**: The degree to which one component depends directly on another
- **cohesion**: How closely related and focused the responsibilities within a single component are
- **dependency injection**: Providing a component's dependencies from outside rather than creating them internally — the primary technique for reducing coupling
- **design pattern**: A named, reusable solution to a commonly recurring structural problem in software architecture[^4]
- **Repository pattern**: Separates data persistence logic from the rest of the application
- **Factory pattern**: Separates object creation logic from the rest of the application
- **Observer pattern**: Separates event producers from event consumers, allowing multiple consumers to react independently
