## Preamble

In Lesson 1, you saw the difference between a `Task` class that did everything and a designed system where each component had one responsibility. But how do you *measure* whether a design is good? How do you explain to a teammate why splitting a class matters beyond "it feels right"?

This lesson gives you three precise tools: **separation of concerns** (what responsibilities belong together), **coupling** (how much components depend on each other), and **cohesion** (how related the elements inside a component are). These are not abstract theory—they are diagnostic metrics you can apply to any codebase today.

By the end of this lesson, you will be able to look at a component diagram and explain why one architecture will age gracefully while another will become brittle. You will also meet three design patterns by name—Factory, Repository, and Observer—that you will see repeatedly throughout this course.

## §1 — Separation of Concerns Revisited

In Lesson 1 §4, we introduced **separation of concerns**—the principle that each component should have one clearly defined responsibility, with minimal overlap into the responsibilities of others.[^1] Let us make this concrete with the Task Management System.

A violation of separation of concerns occurs when a single component handles multiple *types* of work: data storage, user interface formatting, business logic, and external communication all in one place. This is the **God class** pattern you saw in L01 §4.1.

```{mermaid}
classDiagram
    class Task {
        +String title
        +String description
        +String status
        +saveToDatabase()
        +formatForWebDisplay()
        +sendOverdueNotification()
        +assignToUser()
    }
```

The `Task` class above knows about databases, web rendering, notifications, and user assignment. If the notification medium changes (email → SMS), the `Task` class must be modified. If the database schema changes, the `Task` class must be modified. Each change risks breaking unrelated functionality.

Now compare this to a separated design:

```{mermaid}
classDiagram
    class Task {
        +String title
        +String description
        +String status
    }
    class TaskRepository {
        +saveToDatabase(Task)
        +findById(id)
    }
    class TaskFormatter {
        +formatForWebDisplay(Task)
    }
    class NotificationService {
        +sendOverdueNotification(Task)
    }

    TaskRepository --> Task : stores
    TaskFormatter --> Task : formats
    NotificationService --> Task : reads
```

Here, `Task` is pure data. Storage logic lives in `TaskRepository`, UI logic in `TaskFormatter`, and notification logic in `NotificationService`. Each component has one concern.

:::{important}
**The Single Responsibility Test:** Ask "What change would force me to modify this component?" If you can list multiple *unrelated* reasons (database changes, UI changes, notification changes), the component violates separation of concerns.
:::

## §2 — Coupling: Measuring Dependencies

**Coupling** is the degree to which one component depends on another.[^2] Some coupling is unavoidable—components must communicate. The architectural goal is **loose coupling**, where dependencies are minimised and changes to one component have minimal impact on others.

### Tight Coupling

Tight coupling occurs when a component depends directly on the concrete implementation of another component. Changes to the dependency ripple through the system.

```{mermaid}
classDiagram
    class Task {
        +saveToDatabase()
    }
    class MySQLDatabase {
        +connect()
        +execute(query)
    }
    Task --> MySQLDatabase : direct dependency
```

Here, `Task` knows it is talking to `MySQLDatabase`. If you switch to PostgreSQL, you must modify `Task`. This is tight coupling.

### Loose Coupling

Loose coupling introduces an abstraction layer. Components depend on interfaces or contracts, not concrete implementations.

```{mermaid}
classDiagram
    class Task {
        +save()
    }
    class TaskRepository {
        +save(Task)
        +findById(id)
    }
    class MySQLDatabase {
        +connect()
        +execute(query)
    }
    class PostgreSQLDatabase {
        +connect()
        +execute(query)
    }
    
    Task --> TaskRepository : uses
    TaskRepository --> MySQLDatabase : implements with
    TaskRepository ..> PostgreSQLDatabase : can swap to
```

Now `Task` depends on `TaskRepository`, not the database. The repository can switch from MySQL to PostgreSQL without `Task` knowing or caring. This is loose coupling.

:::{warning}
**Coupling disguised as convenience:** It is tempting to call `MySQLDatabase` directly from `Task` because it is fewer lines of code. This convenience accumulates as technical debt—each shortcut makes future changes more expensive.
:::

### Measuring Coupling

| Coupling Level | Description | Change Impact |
|---|---|---|
| **Tight** | Component A knows concrete details of Component B | Changing B requires changing A |
| **Loose** | Component A depends on an abstraction of Component B | Changing B's implementation does not affect A |
| **None** | Components are completely independent | No impact (rare in real systems) |

## §3 — Cohesion: Measuring Internal Relatedness

**Cohesion** is the degree to which the elements within a component belong together.[^3] High cohesion means all the methods and data in a component serve a single, unified purpose. Low cohesion means the component is a grab-bag of unrelated functionality.

### High Cohesion

```{mermaid}
classDiagram
    class TaskRepository {
        +save(Task)
        +findById(id)
        +delete(id)
        +update(Task)
    }
```

All methods in `TaskRepository` relate to one concern: persisting `Task` objects to storage. This is high cohesion.

### Low Cohesion

```{mermaid}
classDiagram
    class TaskUtils {
        +saveToDatabase(Task)
        +formatForWebDisplay(Task)
        +sendEmail(to, subject, body)
        +calculateTax(amount)
        +generatePDF(data)
    }
```

`TaskUtils` handles database operations, UI formatting, email, tax calculation, and PDF generation. These concerns are unrelated. This is low cohesion—a **God class** by another name.

:::{tip}
**The Cohesion Test:** Read the component name aloud: "This component handles X." If X must be a list of unrelated things ("database and email and PDFs and taxes"), cohesion is low.
:::

### Coupling vs. Cohesion: The Relationship

These two metrics work together:

| Scenario | Coupling | Cohesion | Architectural Health |
|---|---|---|---|
| **Ideal** | Loose | High | ✅ Easy to change, easy to understand |
| **Acceptable** | Loose | Low | ⚠️ Easy to change, hard to understand |
| **Fragile** | Tight | High | ⚠️ Hard to change, easy to understand |
| **Disaster** | Tight | Low | ❌ Hard to change, hard to understand |

The goal is **loose coupling + high cohesion**.

```{mermaid}
classDiagram
    class "High Cohesion, Loose Coupling" as Ideal {
        TaskRepository
        TaskFormatter
        NotificationService
    }
    class "Low Cohesion, Tight Coupling" as Disaster {
        Task (does everything)
    }
    
    Ideal --> Disaster : architectural quality spectrum
```

## §4 — Design Patterns: Names for Common Solutions

As you separate concerns and reduce coupling, you will notice certain structural solutions appearing repeatedly. These have names: **design patterns**.[^4] In this course, we name patterns explicitly when they appear so you can recognise them in other codebases.

Three patterns appear naturally in our Task Management System design:

### The Repository Pattern

The **Repository pattern** abstracts data access behind a consistent interface.[^5] Instead of scattering database queries throughout your code, all data access flows through a repository component.

```{mermaid}
classDiagram
    class Task {
        +String title
        +String status
    }
    class TaskRepository {
        +save(Task)
        +findById(id)
        +findAll()
        +delete(id)
    }
    class TaskService {
        +createTask(title)
        +completeTask(id)
    }
    
    TaskService --> TaskRepository : uses
    TaskRepository --> Task : manages
```

`TaskService` never knows how `TaskRepository` stores data. The repository could use MySQL, PostgreSQL, or a flat file—the service does not care. This is the **Repository pattern** you saw in §2.

### The Factory Pattern

The **Factory pattern** is a creational pattern where a function or component creates and returns objects, hiding construction details.[^6]

```{mermaid}
classDiagram
    class Task {
        +String title
        +String status
        +createdAt
    }
    class TaskFactory {
        +createTask(title, assignee)
        +createRecurringTask(title, interval)
    }
    
    TaskFactory --> Task : creates
```

Instead of calling `new Task()` directly throughout your codebase, you call `TaskFactory.createTask()`. The factory handles default values, validation, and initialisation logic. If task creation rules change, only the factory needs modification.

### The Observer Pattern

The **Observer pattern** defines a one-to-many dependency where objects (observers) are notified when another object (subject) changes state.[^7]

```{mermaid}
classDiagram
    class TaskEventPublisher {
        +publish(event)
        +subscribe(observer)
    }
    class NotificationService {
        +onTaskOverdue(event)
    }
    class AuditLogger {
        +onTaskCreated(event)
    }
    class TaskMetrics {
        +onTaskCompleted(event)
    }
    
    TaskEventPublisher --> NotificationService : notifies
    TaskEventPublisher --> AuditLogger : notifies
    TaskEventPublisher --> TaskMetrics : notifies
```

When a task event occurs (created, completed, overdue), the `TaskEventPublisher` notifies all subscribed observers. Adding a new observer (e.g., `SlackNotifier`) does not require modifying the publisher or existing observers. This is the **Observer pattern**.

:::{note}
**Gang of Four reference:** These three patterns are documented in *Design Patterns: Elements of Reusable Object-Oriented Software* (Gamma, Helm, Johnson, Vlissides — 1994), commonly called the "Gang of Four" book. You do not need to read it for this course, but the names we use come from this source.
:::

## Summary

| Concept | What you learned | Why it matters |
|---|---|---|
| Separation of concerns | Each component should have one clearly defined responsibility | Prevents God classes and makes changes isolated |
| Coupling | The degree to which components depend on each other | Loose coupling makes systems easier to modify |
| Cohesion | The degree to which elements within a component belong together | High cohesion makes components easier to understand |
| Repository pattern | Abstracts data access behind a consistent interface | Decouples business logic from storage implementation |
| Factory pattern | Encapsulates object creation logic | Centralises construction rules and defaults |
| Observer pattern | Enables one-to-many state change notifications | Allows new behaviours without modifying existing code |

---

## Key Terms

- **separation of concerns**: The principle that each component should have one clearly defined responsibility, with minimal overlap into the responsibilities of others
- **coupling**: The degree to which one component depends on another
- **cohesion**: The degree to which the elements within a component belong together
- **loose coupling**: A design where components depend on abstractions, not concrete implementations
- **high cohesion**: A design where all elements in a component serve a unified purpose
- **God class**: A component that has absorbed too many responsibilities—doing data storage, presentation, business logic, and communication all at once
- **Repository pattern**: A pattern that abstracts data access behind a consistent interface
- **Factory pattern**: A creational pattern where a function creates and returns objects, hiding construction details
- **Observer pattern**: A pattern where objects (observers) are notified when another object (subject) changes state
- **module boundary**: The defined interface between two components—what is exposed vs. what is hidden
